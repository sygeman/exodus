# Exodus Control

Слой планирования, управления задачами и оркестрации изменений.
Control работает только с уже стабилизированным Intent, полученным из Intent Service.
Он не интерпретирует смысл и не анализирует ввод пользователя.

---

## Планирование (ACSD)

Control формирует план выполнения на основе Intent:

* декомпозиция задачи на ACSD
* выбор стратегии реализации
* разбиение на Tasks и подзадачи
* определение зависимостей между шагами
* выбор инструментов и моделей из Infra

---

## Управление задачами и проектами

Control является единственным слоем, который управляет жизненным циклом проектов и задач:

* создает и ведет Projects
* создает и ведет Tasks
* строит Task Graph (зависимости между задачами)
* связывает Task с ACSD
* связывает Task с git веткой и коммитами
* отслеживает состояние Task:
  draft → active → in_progress → stabilized → merged → closed
* гарантирует изоляцию задач друг от друга через ветки и snapshot State

---

## Связь с State

Control управляет синхронизацией с git как единственным источником состояния:

* создает ветки под Task или группу Tasks (trunk-based flow)
* поддерживает ветки до стабилизации ACSD
* инициирует merge в main после стабилизации
* связывает commits с Task и ACSD слоями
* синхронизирует изменения кода и ACSD yaml
* обеспечивает целостность соответствия “ACSD ↔ код ↔ история изменений”

State не изменяется напрямую, только через Control -> Execution -> State.

---

## Execution orchestration

Control управляет запуском Execution Runtime:

* инициирует Execution только для Task в активном состоянии
* передает snapshot State, ограниченный задачей
* задает allowed skills и ограничения среды
* получает результат в виде diff и artifacts
* валидирует результат выполнения
* принимает решение: повторный запуск, корректировка Task или фиксация в State

Execution не имеет доступа к Control или глобальному State.

---

## Протокол перед Execution

Перед запуском Execution Control обязан:

* показать, что система поняла (Intent → ACSD snapshot)
* показать, что система собирается сделать (Task plan + затронутые части ACSD + изменения в State)
* дождаться явного подтверждения пользователя

---

## Жесткие ограничения

Control не имеет права:

* интерпретировать сырой пользовательский ввод (это делает Intent Service)
* менять State без Task
* запускать Execution без Task
* изменять Intent после его стабилизации (только через новый Intent → новый Task)
* выполнять код напрямую

---

## Итоговая роль

Control = stateful оркестратор жизненного цикла изменений системы

Он управляет переходом:

Intent (внешний сервис) → ACSD Plan → Tasks → Execution → Git State

---

## Суть системы

Control — это точка, где зафиксированная идея превращается в управляемую серию изменений в коде и структуре продукта.
