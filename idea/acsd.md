# ACSD — AI-Cascading Solo Development

---

## 1. СУТЬ

Человек не редактирует файлы.
Человек формулирует намерения на естественном языке.
Система генерирует, валидирует и поддерживает все артефакты каскада — от Vision до кода.

Логика не теряется, потому что:
1. Каждая сущность имеет явный ID и ссылку на родителя
2. Валидация проходит по всей цепочке до L0, а не только до соседа
3. Последствия изменений показываются до начала пересборки

---

## 2. КАСКАД УРОВНЕЙ

Уровень | Артефакт                    | Назначение                          | Ссылка
--------|-----------------------------|-------------------------------------|-------------
L0      | acsd/vision.md              | Цель, границы, инварианты [V-##]    | —
L1      | acsd/architecture.md        | Компоненты, стек, потоки [A-##]     | covers: V-##
L2      | modules/*/spec.md           | Ответственность модуля [S-##]       | covers: A-##
L3      | modules/*/contract.md       | Публичный API, типы [C-##]          | covers: S-##
L4      | modules/*/__tests__/*.test  | Контрактные тесты                   | implements: C-##
L5      | modules/*/*.ts              | Реализация                          | implements: C-##

Правило: Уровень N+1 не существует без уровня N. Изменение N автоматически инвалидирует все зависимые уровни ниже.

## 3. ТРАССИРОВКА

### 3.1. Формат ID в контенте

Пример для vision.md:
- [V-01] Пользователи аутентифицируются через биометрию
- [V-02] Данные шифруются на rest

Пример для architecture.md:
- [A-03] Модуль auth реализует биометрическую аутентификацию
  covers: V-01
- [A-04] AES-256 для хранения чувствительных данных
  covers: V-02

### 3.2. Frontmatter для машинного парсинга

Каждый файл уровня L2-L5 содержит заголовок:
```
---
id: S-02
level: 2
parents: [A-03]
children: [C-01, C-02]
status: active
---
```
Зачем оба формата:
- ID в тексте — для человека и LLM (семантика + читаемость)
- Frontmatter — для оркестратора (быстрый парсинг, индексация, кэш)

## 4. ВАЛИДАЦИЯ: ДВУХФАЗНАЯ, СКВОЗНАЯ ДО L0

### Фаза 1: Механическая проверка (быстрая)

Проверяет целостность ссылок в графе:
1. Все родители существуют
2. Все дети существуют (или помечены @deferred)
3. Нет висячих ссылок

### Фаза 2: Семантическая проверка через LLM (полная цепочка)

LLM получает контент всей цепочки от L0 до L5 и отвечает на вопросы:
1. Соответствует ли код исходному видению (L0)?
2. Есть ли накопленные противоречия в цепочке?
3. Где первый разрыв логики?

Ответ возвращается в формате JSON:
```
{
  "valid": boolean,
  "issues": [{"level": number, "message": string, "suggestion": string}],
  "first_break": string | null
}
```

Правило: Если валидация не пройдена — артефакт не фиксируется, система предлагает исправление.

## 5. РОЛИ

### 5.1. Человек — Критик

Интерфейс: естественный язык (чат, голос, комментарии).

Делает:
- Формулирует намерения
- Оценивает сгенерированные артефакты
- Принимает решения перед фиксацией
- Запрашивает альтернативы

Не делает:
- Не открывает редактор файлов
- Не правит код, спеки, контракты
- Не отслеживает связи вручную
- Не держит архитектуру в голове

### 5.2. LLM — Трансформер + Валидатор

Делает:
- Генерация артефакта уровня N+1 из уровня N
- Семантическая валидация цепочки до L0
- Предложение альтернатив при конфликтах

Ограничения:
- Не добавляет сущности без обоснования в родителях
- Не принимает архитектурных решений
- Работает с temperature=0, кэш по хэшу входа
- Не фиксирует изменения без подтверждения человека

### 5.3. Оркестратор — Движок

Задачи:
- Парсинг артефактов: извлечение ID из контента + frontmatter
- Построение и поддержка графа зависимостей (.acsd/graph.json)
- Запуск макроцикла (BFS сверху вниз) и микроциклов
- Кэширование LLM-запросов по хэшу
- Двухфазная валидация: механическая + семантическая
- Анализ влияния изменений: расчёт затронутых узлов до пересборки

Скрытое состояние (.acsd/):
- graph.json — граф зависимостей
- cache/ — кэш LLM-запросов
- state.json — состояние сборки
- logs/ — история валидаций и генераций

Пользователь не взаимодействует с этой папкой напрямую.

## 6. МАКРОЦИКЛ

1. Загрузить граф из .acsd/graph.json
2. Построить очередь узлов в порядке BFS от L0
3. Для каждого узла:
   а) Если артефакт отсутствует — запустить микроцикл генерации
   б) Если хэш родителя изменился — пометить на перегенерацию
   в) Если артефакт изменён — валидировать цепочку
4. После генерации/обновления:
   а) Механическая валидация ссылок
   б) Семантическая валидация полной цепочки до L0 через LLM
   в) При ошибке — предложить исправление, ждать подтверждения
5. Повторять до фиксированной точки (нет изменений)
6. Сохранить обновлённый граф и состояние

Свойства:
- Идемпотентность: повторный запуск даёт одинаковый результат
- Инкрементальность: пересобираются только затронутые поддеревья
- Безопасность: ни один артефакт не фиксируется без прохождения валидации

---

## 7. МИКРОЦИКЛ: ГЕНЕРАЦИЯ ОДНОГО АРТЕФАКТА

Вход:
- Родитель: контент + метаданные уровня N
- Шаблон: структура артефакта уровня N+1
- Контекст: смежные файлы того же уровня (опционально)
- Ограничения: inherited constraints из цепочки до L0

Процесс:
1. Сформировать промпт с инструкцией развернуть уровень N+1 из N
2. Вызвать LLM с temperature=0, проверить кэш по хэшу входа
3. Пост-обработка: вставить frontmatter, нормализовать формат
4. Валидация: механическая + семантическая
5. При провале (макс. 3 попытки) — уточнить промпт или запросить фидбек

Выход:
- Сгенерированный файл уровня N+1
- Метаданные для графа: id, hash, parents, children, status

## 8. АНАЛИЗ ВЛИЯНИЯ ИЗМЕНЕНИЙ

Перед применением любого изменения система показывает:

1. КОНФЛИКТЫ С ТЕКУЩИМ VISION
   - Какие требования противоречат новому намерению
   - Варианты решения (заменить/удалить/изменить)

2. АРХИТЕКТУРНЫЕ ОГРАНИЧЕНИЯ
   - Какие решения становятся невалидными
   - Требуемые изменения в ADR

3. ВЛИЯНИЕ НА ПРОЕКТ
   - Затронутые модули
   - Затронутые артефакты (по уровням)
   - Оценка времени пересборки и ручной проверки

4. ВАРИАНТЫ ДЕЙСТВИЙ
   - Принять изменения (пересобрать всё)
   - Изменить намерение (убрать конфликт)
   - Запросить альтернативу (компромиссный вариант)
   - Отложить (сохранить текущее состояние)

Правило: Система не начинает пересборку без явного подтверждения человека.

## 9. ИНТЕРФЕЙС ВЗАИМОДЕЙСТВИЯ

### Естественный язык (основной)

Человек: Хочу сервис для заметок с тегами и офлайн-синхронизацией

Система:
- Генерирует vision.md с [V-01] Заметки, [V-02] Теги, [V-03] Офлайн
- Запускает макроцикл до кода
- Показывает: Сгенерировал проект. Что думаешь?

Человек: Офлайн-синхронизация слишком сложная, упрости до сохранить локально

Система:
- Обновляет [V-03], пересчитывает влияние
- Перегенерирует затронутые модули
- Показывает диффы, ждёт подтверждения

Человек: Ок

Система:
- Фиксирует изменения в графе
- Сообщает: Готово. Всё согласовано.

### Команды CLI (опционально)
```
acsd init "Описание идеи"       — Создать проект из намерения
acsd review                     — Показать последние изменения
acsd check                      — Полная валидация (механическая + семантическая)
acsd trace [ID]                 — Показать цепочку от сущности до L0
acsd impact [ID]                — Показать влияние изменения до применения
acsd rebuild [ID]               — Принудительная перегенерация узла и потомков
acsd alternative [ID]           — Запросить альтернативный вариант
acsd accept [ID]                — Подтвердить черновик, зафиксировать в графе
```

## 10. СТРУКТУРА ПРОЕКТА
```
project/
  acsd/
    vision.md                   — L0: видение [V-##]
    architecture.md             — L1: архитектура [A-##]
    adr/                        — Architectural Decision Records

  modules/
    {name}/
      spec.md                   — L2: спецификация [S-##] + frontmatter
      contract.md               — L3: контракт [C-##] + frontmatter
      __tests__/
        *.test.ts               — L4: контрактные тесты
      src/
        *.ts                    — L5: реализация

  .acsd/                        — Скрытое состояние (не редактировать)
    graph.json                  — Граф зависимостей
    cache/                      — Кэш LLM-запросов
    state.json                  — Состояние сборки

  acsd.config.json              — Настройки: модель, шаблоны, правила
  package.json                  — Скрипты: build, sync, check
  README.md
```
Важно: Пользователь не редактирует файлы в modules/ напрямую. Только через обратную связь.

## 11. ОБРАБОТКА ИСКЛЮЧЕНИЙ

### Временный дрейф (EXCEPTION)

Файл может содержать исключение с ограничением по времени:

---
id: S-02
exceptions:
  - id: EX-01
    reason: Критический баг в production, требуется хотфикс
    valid_until: 2024-12-31
    approved_by: human
---

Правила:
- Исключение должно быть явно помечено и ограничено по времени
- Система напоминает об истечении valid_until
- После истечения — блокировка, требуется нормализация через каскад

### Архитектурные решения (ADR)

ADR создаётся при конфликте требований:

# acsd/adr/ADR-001-biometric-auth.md

Статус: Accepted

Контекст:
Требуется биометрическая аутентификация, но [V-05] разрешает только пароль.

Решение:
Заменить [V-05] на мультифакторная аутентификация (пароль + биометрия).

Последствия:
- [A-01] обновлён: разрешён биометрический провайдер X
- Затронутые модули: auth, user, settings

Правило: ADR создаётся системой при конфликте, человек подтверждает, система применяет изменения каскадом.

## 12. ОГРАНИЧЕНИЯ ПОДХОДА

Ограничение                          | Обоснование
-------------------------------------|------------------------------------------
Не для хаотичного прототипирования   | Каскад требует осознанных намерений
Не для срочных хотфиксов без EXCEPTION | Изменения должны идти сверху вниз
Требует чёткой формулировки идей     | Система не угадывает, а трансформирует
Сквозная валидация медленнее локальной | Гарантия целостности важнее скорости

Это не недостатки. Это плата за гарантию, что логика не потеряется.

[ ] Система поддерживает EXCEPTION и ADR для управляемых исключений

## 14. МИНИМАЛЬНАЯ РЕАЛИЗАЦИЯ (MVP)

1. Парсер артефактов — извлекает [ID] из контента + frontmatter
2. Граф зависимостей — хранит связи, хэши, статусы (.acsd/graph.json)
3. LLM-обёртка — промпты, кэш по хэшу, temperature=0, JSON-ответы
4. Валидатор ссылок — механическая проверка целостности графа
5. Валидатор цепочки — LLM-проверка смысла от кода до Vision
6. Анализатор влияния — расчёт затронутых узлов при изменении
7. Три команды: init, check, review

Всё остальное — оптимизации и расширения.
