# Module Architecture

module: <module-name>

Описание внутренней структуры модуля и правил его реализации.  
Не раскрывает глобальную архитектуру или другие модули напрямую.

---

## 1. Module Responsibility

Краткое описание роли модуля.

Пример:

Модуль управляет пользователями системы.  
Он отвечает за создание, чтение и обновление профилей.  
Не занимается аутентификацией или хранением сессий.

---

## 2. Internal Components

Список внутренних компонентов и их назначение.

| component | responsibility |
|-----------|----------------|
| service | бизнес-логика |
| repository | доступ к persistence (через интерфейс storage) |
| validator | проверка входных данных |
| notifier | генерация событий / логирование |

---

## 3. Component Interactions

Как внутренние компоненты взаимодействуют.

| from      | to         | purpose |
|-----------|------------|---------|
| service   | repository | хранение и извлечение данных |
| service   | validator  | валидация input |
| service   | notifier   | уведомления |

Правило: **только внутри модуля**.  
Внешние модули доступны через их interface.md.

---

## 4. Data Flow

Краткое описание движения данных внутри модуля.

Пример:

- входящие DTO → validator → service → repository → output DTO  
- service генерирует события через notifier

---

## 5. Module Invariants

Инварианты, которые нельзя нарушать:

- все операции должны быть детерминированы  
- внутренние компоненты используют DTO из global data model  
- repository не раскрывает внутренние структуры наружу  
- все ошибки соответствуют списку Error Codes

---

## 6. Integration Points

Внешние зависимости и их использование:

| module | usage |
|--------|-------|
| storage | хранение пользователей |
| logger  | запись логов |

---

## 7. Evolution Rules

Что разрешено менять:

- добавление новых внутренних компонентов
- расширение функциональности через новые операции

Что запрещено:

- нарушение invariants
- обход interface других модулей
- изменение глобальной модели данных
