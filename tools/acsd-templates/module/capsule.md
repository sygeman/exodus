# Module Capsule

Module: <module name>

Capsule содержит контекст, необходимый для корректной реализации модуля.

Этот документ предназначен для AI.

---

## 1. Module Purpose

Краткое объяснение назначения модуля.

Пример:

Модуль управляет пользователями системы.
Он отвечает за создание, получение и обновление профилей.

Модуль не занимается аутентификацией или хранением сессий.

---

## 2. Domain Concepts

Основные понятия домена.

| concept | description |
|------|-------------|
| user | зарегистрированный пользователь |
| profile | публичные данные пользователя |
| account | учётная запись |

---

## 3. Business Rules

Ключевые правила домена.

- email пользователя уникален
- пользователь может иметь только один аккаунт
- удаление пользователя может быть запрещено

---

## 4. Implementation Constraints

Ограничения реализации.

Пример:

- email должен храниться в нормализованном виде
- операции должны быть идемпотентными
- ошибки должны возвращаться из списка interface.md

---

Какие данные принадлежат модулю.

| entity | ownership |
|------|-----------|
| UserDTO | owner |
| SessionDTO | external |

## 6. Dependencies Usage

Как модуль использует зависимости.

| module | usage |
|------|------|
| storage | хранение пользователей |

- зависимости используются только через их интерфейс

---

## 7. Edge Cases

Сложные случаи.

Примеры:

- пользователь с таким email уже существует
- пользователь удалён
- обновление пользователя с устаревшими данными

---

## 8. Non Goals

Что модуль не должен делать.

- аутентификация
- управление сессиями
- обработка HTTP

---

## 9. Known Risks

Потенциальные проблемы.

- дублирование email
- гонки при обновлении данных

---

## 10. Implementation Notes

Полезные подсказки для реализации.

Пример:

- email лучше индексировать
- операции должны быть атомарными
