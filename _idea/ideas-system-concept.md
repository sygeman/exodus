# Концепция системы идей (Ideas System)

## История обсуждения

### 1. От файлов к связям
- Было: фокус на типах файлов (vision/design/implementation)
- Стало: фокус на связях между идеями
- Ключевой инсайт: не важно ГДЕ лежит файл, важно КАК идеи связаны между собой

### 2. От тезисов к идеям
- Было: "тезисы" (theses) — академично, статично, напоминает дискуссию/спор
- Стало: "идеи" (ideas) — интуитивно, гибко, подходит для продуктового контекста
- Каждая идея — атомарная мысль с типом и связями

### 3. Уровни детализации
- Уровни (L0/L1/L2/L3) остаются как мета-атрибут файла
- Не жёсткая структура директорий, а базовый уровень для идей в файле
- Связи могут вести на любой уровень (L1 может ссылаться на L0, L2, или другой L1)

### 4. Полная цепочка: Идея → Продукт

```
Идеи (в голове)
    ↓
[Фиксация] — идеи в системе
    ↓
[Vision] — стабилизированное намерение
    ↓
[Design] — архитектура и компоненты
    ↓
[Specification] — детальные требования
    ↓
[Implementation Plan] — план реализации
    ↓
[Code] — исполняемая система
    ↓
[Tests] — проверка корректности
    ↓
[Documentation] — инструкции для пользователей
    ↓
[Deployment] — работающий продукт
```

Код — это только один из артефактов в цепочке превращения идеи в продукт.

## Структура системы

### Минимальная единица — Idea

```yaml
ideas:
  - id: V-01                    # Уникальный идентификатор
    type: capability            # Тип идеи
    text: Описание идеи         # Содержание (человекочитаемое)
    requires: [V-02]            # Связи с другими идеями
    supports: [INV-01]          # Множественные связи разных типов
```

### Типы идей

| Тип | Описание | Пример |
|-----|----------|--------|
| `capability` | Возможность системы | "Превращать хаотичные мысли..." |
| `constraint` | Ограничение | "Работает только с git" |
| `invariant` | Неизменное правило | "Намерение стабилизируется перед..." |
| `principle` | Принцип проектирования | "Модульность" |
| `non_goal` | То, чего не будет | "Не заменяет IDE" |
| `component` | Компонент системы | "Intent Layer" |
| `decision` | Принятое решение | "Использовать YAML" |
| `observation` | Наблюдение | "Пользователи жалуются..." |
| `insight` | Инсайт | "80% бросают на шаге 3" |
| `assumption` | Допущение | "Retention вырастет если..." |

### Типы связей

| Связь | Описание | Направление |
|-------|----------|-------------|
| `requires` | Требует наличия | A требует B |
| `supports` | Поддерживает/обосновывает | A поддерживает B |
| `contradicts` | Противоречит | A противоречит B |
| `derives_from` | Производно от | A выводится из B |
| `implements` | Реализует | A реализует B |
| `enables` | Делает возможным | A делает возможным B |
| `part_of` | Часть целого | A является частью B |

## Пример структуры файла

```yaml
# exodus/vision.yaml
meta:
  level: L0
  type: vision
  id: exodus-vision

ideas:
  - id: V-01
    type: capability
    text: Превращать хаотичные мысли в стабилизированное намерение
    
  - id: V-02
    type: capability
    text: Детализировать намерение через каскад уровней
    requires: [V-01]
    
  - id: INV-01
    type: invariant
    text: Намерение должно быть стабилизировано перед планированием
    supports: [V-01, V-02]
    
  - id: NG-01
    type: non_goal
    text: Не является генератором кода без контекста
    contradicts: [V-03]
```

```yaml
# exodus/design/intent.yaml
meta:
  level: L1
  type: design
  id: exodus-design-intent

ideas:
  - id: D-INTENT-01
    type: component
    text: Intent Layer — интерпретация смысла
    implements: V-01
    
  - id: D-INTENT-02
    type: mode
    text: Discovery Mode — исследование и накопление идей
    part_of: D-INTENT-01
    derives_from: V-01
    
  - id: D-INTENT-03
    type: mode
    text: Stabilization Mode — фиксация намерения
    part_of: D-INTENT-01
    derives_from: V-01
    requires: [D-INTENT-02]
```

## Граф связей

Из примеров выше строится граф:

```
V-01 ← implements ← D-INTENT-01
  ↑                    ↑
  └─ supports ─ INV-01  └─ part_of ─ D-INTENT-02
                              ↑
                              └─ derives_from ─ V-01

V-02 ← requires ─ V-01
  ↑
  └─ supports ─ INV-01
```

## Принципы системы

1. **Файл — контейнер** для идей одного базового уровня
2. **Связи определяют семантику**, не локация файла
3. **Каждая идея должна иметь связь** (нет orphaned идей)
4. **Типы связей типизированы** — не просто "связь", а конкретное отношение
5. **Система строит сама себя** (dogfooding) — Exodus описывает Exodus

## Валидационные правила (предполагаемые)

- Нет orphaned идей — у каждой идеи есть хотя бы одна связь
- Нет циклов в графе — A → B → C → A запрещено
- Нет противоречий — если A contradicts B, то B не может supports A
- Уровни консистентны — L1 идеи ссылаются в основном на L0 и L1

## Открытые вопросы

1. **Валидация**: строгая (ошибка) или мягкая (предупреждение)?
2. **Расширяемость типов**: фиксированный набор или можно добавлять свои?
3. **Версионирование**: как отслеживать эволюцию идей?
4. **Визуализация**: нужен ли графический просмотр связей?
5. **Интеграция с кодом**: как связать идеи с конкретными файлами кода?
