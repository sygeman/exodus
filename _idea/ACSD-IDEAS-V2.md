# ACSD Ideas System — v2

### Формат
- Чистый YAML — все артефакты
- Файл — контейнер идей одного базового уровня
- Связи определяют семантику, не локация файла
- Машинная читаемость приоритет

### Каскад
- Каждый уровень — детализация предыдущего
- Все уровни обязательны
- Уровень N+1 не существует без N
- Изменение N автоматически инвалидирует зависимые уровни

### Валидация
- Строгая — орфаны, циклы, противоречия, висячие ссылки = error
- Двухфазная:
  - Фаза 1: механическая (целостность графа)
  - Фаза 2: LLM семантическая (соответствие кода vision, поиск разрывов логики)
- Валидация начинается с L0 — любая идея проверяется через полную цепочку до Vision
- Каждая идея может ссылаться только на N-1 (родитель) или N (сосед), пропуск уровней запрещён
- Ни один артефакт не фиксируется без подтверждения человека

### Граф
- Связи — first-class citizens
- Каждая идея обязана иметь хотя бы одну связь
- Нет циклов, нет противоречий
- ADR и EXCEPTION для управляемых отклонений

### Роли
- Человек — формулирует намерения, оценивает, принимает решения
- LLM — генерирует артефакты, валидирует семантику, предлагает альтернативы
- Система — парсит, строит граф, запускает каскад, анализирует влияние

### Ограничения LLM
- temperature=0
- Кэш по хэшу входа
- Не добавляет сущности без обоснования в родителях
- Не принимает архитектурных решений

### Набор уровней каскада

ACSD начинается с L0 Vision. Intent — это слой до ACSD (Discovery + Stabilization), результат которого попадает в ACSD как L0.

Итоговый каскад — 5 уровней, все декларативные:

| Уровень | Артефакт | Назначение |
|---------|----------|------------|
| L0 | Vision | Цель, границы, инварианты |
| L1 | Design | Компоненты, их ответственность, связи |
| L2 | Specification | Требования к модулю, сущности, операции |
| L3 | Contract | Публичный API, типы, сигнатуры, ошибки |
| L4 | Code | Реализация + тесты + конфиги |


### Модель связей

5 типов связей

| Тип | Отношение | Направление | Пример |
|-----|-----------|-------------|--------|
| implements | детализация | сверху вниз | Contract implements Specification |
| requires | зависимость | горизонтальное | Auth requires Storage |
| part_of | композиция | часть → целое | Module part_of System |
| supports | обоснование | - | Metric supports Goal |
| contradicts | конфликт | - | Fast contradicts Secure |

### Обоснование решений

Каждый выбор (type: decision) обязан иметь полное обоснование

Структура:
```yaml
- id: D-01
  type: decision
  text: "Использовать PostgreSQL"
  implements: S-01
  alternatives:
    - option: "MongoDB"
      pros: ["Гибкая схема"]
      cons: ["Нет ACID"]
      rejected_because: "Нет ACID для финансов"
    - option: "PostgreSQL"
      pros: ["ACID", "Команда знает"]
      cons: ["Нужны миграции"]
      selected: true
```

Проверки валидатора:
- Ровно одна альтернатива с `selected: true`
- У невыбранных альтернатив обязательно поле `rejected_because`

### Путь к корневому файлу

`exodus.yaml` обязателен в корне проекта. Поле `vision` указывает путь к L0.

```yaml
# exodus.yaml
vision: ./exodus/vision.yaml
```

- Если `vision` не указан — ошибка конфигурации
- Путь может быть любым, рекомендуемая конвенция — `./exodus/vision.yaml`


### Типы идей

| Тип | Правила валидации | Пример |
|-----|-------------------|--------|
| goal | Должен быть достижим | "Пользователи создают заметки" |
| non_goal | Явно исключено из scope | "Не делать собственную БД" |
| constraint | Может измениться | "Работает только офлайн" |
| invariant | Неизменен, EXCEPTION только с valid_until | "Данные принадлежат пользователю" |
| component | Должен иметь part_of или contains | "Модуль notes" |
| decision | Обязан иметь alternatives | "Выбрали PostgreSQL" |
| principle | Поддерживает goal, не детализирует | "Простота превыше оптимизации" |
